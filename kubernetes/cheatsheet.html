Control Plane & Node

  • kube-apiserver — front door: authN/authZ/admission, persists to etcd
  • etcd — consistent HA key-value store; cluster's single source of truth
  • kube-scheduler — picks a node for each unbound Pod
  • kube-controller-manager — bundled control loops (ReplicaSet, Node, Job...)
  • cloud-controller-manager — cloud integration: LB provisioning, Node labels/IPs
  • kubelet — runs on every node; starts/stops containers via CRI, runs static Pods
  • kube-proxy — programs per-node Service dataplane (iptables or IPVS)

RBAC

  • Role — namespaced rules only; can't touch Nodes/PVs or cluster-wide scope
  • ClusterRole — needed for cluster-scoped resources, non-resource paths, or one rule set applied across every namespace
  • RoleBinding — grants a Role or ClusterRole, scoped to its own namespace only
  • ClusterRoleBinding — grants a ClusterRole cluster-wide, no namespace boundary
  • Rules are purely additive — no deny rule exists
  • Creating a broader role/binding needs escalate/bind verbs or already holding those perms

CNI / CRI / CSI

  • CRI — gRPC iface kubelet→runtime (containerd, CRI-O); no default runtime shipped
  • CNI — Pod networking plugin; no default shipped, CoreDNS stays Pending without one
  • CSI — out-of-tree storage driver interface; replaced in-tree volume plugins

Services

  • ClusterIP (default) — internal VIP; clusterIP: None = headless, DNS returns Pod IPs directly
  • NodePort — every node listens on one port, range 30000–32767 by default
  • LoadBalancer — cloud LB provisioned async, usually layers on a NodePort
  • EndpointSlices track ready backends; new slice auto-starts ~every 100 endpoints
  • Ingress needs a controller + IngressClass; TLS only on port 443; Gateway API is the recommended successor (GatewayClass/Gateway/HTTPRoute)

Storage

  • PV — cluster-scoped real storage; PVC — namespaced request; StorageClass — named provisioner profile
  • Access modes: RWO one node r/w · ROX many read-only · RWX many r/w · RWOP one Pod only
  • Reclaim: Retain keep PV+data (manual cleanup) · Delete removes both (dynamic default) · Recycle deprecated
  • volumeBindingMode: WaitForFirstConsumer delays binding until a Pod is scheduled (vs. Immediate default)
  • allowVolumeExpansion: true — grow only, never shrink

Probes

  • Liveness — fail → container restarted (app hung/deadlocked)
  • Readiness — fail → removed from Service endpoints, no restart
  • Startup — holds off liveness/readiness until it first succeeds (slow boot)

Requests, Limits & QoS

  • Request — scheduler's placement floor; node must have that much unallocated
  • Limit — CPU: throttled at ceiling · Memory: OOMKilled by kernel at ceiling
  • Guaranteed — every container sets request = limit for CPU & memory
  • Burstable — some request/limit set, doesn't meet Guaranteed bar
  • BestEffort — no request/limit at all; first killed under memory pressure

Scheduling Controls

  • Taint repels; toleration permits: NoSchedule blocks new · PreferNoSchedule soft · NoExecute also evicts running
  • Unreachable node auto-tainted NoExecute+NoSchedule; default 300s tolerance before eviction
  • nodeAffinity ops: In/NotIn/Exists/DoesNotExist/Gt/Lt
  • requiredDuringScheduling... = hard · preferred... = weighted 1–100 soft
  • Pod affinity/anti-affinity uses the same required/preferred vocabulary, relative to co-located Pods

Rollouts

  • RollingUpdate defaults: maxUnavailable 25% (rounds down), maxSurge 25% (rounds up)
  • Recreate — kills all old Pods first, gap with zero running
  • Only Pod template changes trigger a rollout; scaling alone does not
  • progressDeadlineSeconds 600s default → stalled rollout flagged
  • revisionHistoryLimit 10 default caps how far back rollback can reach

kubectl Troubleshooting

  • describe pod/node — Events section names the actual blocker
  • logs -c <c> --previous — prior crashed container's output
  • exec needs a shell in-image; debug -it --target= attaches an ephemeral container sharing the process namespace instead
  • debug node/<n> — SSH-free node shell, host fs at /host
  • top nodes/pods needs metrics-server; describe node shows requested vs. top's actual usage
  • drain (evict) → maintain → uncordon; never skip drain before reset/delete
