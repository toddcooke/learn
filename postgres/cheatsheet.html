MVCC, VACUUM & ANALYZE

  • Readers never block writers, writers never block readers — an UPDATE/DELETE adds a new row version instead of overwriting
  • Snapshot = "committed as of" a moment; Read Committed takes one per statement, Repeatable Read/Serializable take one per transaction
  • VACUUM reclaims dead-tuple space for reuse but never shrinks the file
  • VACUUM FULL rewrites the table (shrinks it); needs ACCESS EXCLUSIVE + ~2× disk space; autovacuum never runs it
  • ANALYZE refreshes planner stats (pg_statistic/pg_stats) — a different job from VACUUM, neither substitutes for the other
  • Autovacuum fires when dead tuples > autovacuum_vacuum_threshold (50) + scale_factor (20%) × table rows
  • TRUNCATE / rewriting ALTER TABLE bypass MVCC — a snapshot that never touched the table can see it go empty

Read Uncommitted is silently treated as Read Committed (no dirty reads possible under MVCC). *PG's Repeatable Read blocks phantoms too — stricter than the SQL standard requires. Repeatable Read/Serializable conflicts abort with SQLSTATE 40001 (retry the whole transaction).

Locking

  • 8 table lock modes, weakest→strongest: ACCESS SHARE (SELECT) … ROW EXCLUSIVE (INSERT/UPDATE/DELETE) … ACCESS EXCLUSIVE (DROP/TRUNCATE/VACUUM FULL) — only ACCESS EXCLUSIVE blocks a plain SELECT
  • Row locks, strongest→weakest: FOR UPDATE > FOR NO KEY UPDATE > FOR SHARE > FOR KEY SHARE
  • Deadlock: auto-detected after deadlock_timeout (1s default), aborts one transaction — SQLSTATE 40P01
  • Advisory locks: session-scoped (held until released or disconnect) vs transaction-scoped (auto-released at commit)

Partial & Expression Indexes

  • Partial index only helps if the planner can prove the query's WHERE implies the index predicate at plan time — a parameterized clause (x < $1) defeats this
  • Unique partial index enforces uniqueness over a subset of rows, or "at most one NULL" (ordinary UNIQUE never rejects a 2nd NULL)
  • Expression index (e.g. lower(email)) recomputes on every write, not at query time
  • Many category-specific partial indexes usually lose to one composite index, or to real partitioning at scale

EXPLAIN & Scan Types

  • Plain EXPLAIN never executes the statement — zero side effects, even for INSERT/UPDATE/DELETE
  • EXPLAIN ANALYZE genuinely runs it (real writes happen); wrap in BEGIN;…ROLLBACK; to inspect without keeping effects
  • Selectivity ladder: Seq Scan (little/no filter) → Bitmap Index/Heap Scan (moderately selective) → Index Scan (highly selective / point lookup)
  • cost=startup..total rows=N width=W — rows is estimated output, not rows examined

WAL & Checkpoints

  • A WAL record must be flushed before its data page is allowed to hit disk — durability backbone; crash recovery replays WAL forward (REDO) from the last checkpoint
  • Only the sequential WAL needs an fsync at commit, not every scattered data page
  • A checkpoint flushes all dirty pages; a small max_wal_size forces checkpoints more often
  • Archiving/PITR needs wal_level ≥ replica + archive_mode on + an archive_command

Key Config Knobs

  • shared_buffers — start ~25% of RAM; rarely helps much beyond ~40% of RAM (OS page cache does the rest)
  • work_mem — PER-OPERATION, not per-connection; caps one sort/hash before it spills to disk; many ops × many sessions can multiply total use far past the configured value
  • max_connections — each session is a full backend process, not a thread; raising it raises baseline memory reserved even before any query runs — use an external pooler (PgBouncer) instead
  • autovacuum_vacuum_threshold 50 rows + scale_factor 20% of table rows

Replication Modes

  • Physical (streaming) — byte-identical, whole-cluster, standby must match primary's version
  • Logical — row-level pub/sub via replication identity; selective tables, cross-major-version OK
  • Default is async: commit doesn't wait for the standby. Opt-in sync via synchronous_standby_names; synchronous_commit levels: remote_write < on (flushed) < remote_apply (visible to queries)
  • Hot standby replays WAL + serves read queries; warm standby replays only, no client connections
  • A cascading standby feeds further standbys downstream; always async regardless of the primary's sync settings

Backup & Recovery

  • pg_dump — one database, logical, plain SQL or archive format (pairs with pg_restore)
  • pg_dumpall — adds cluster-wide roles/tablespaces/param grants pg_dump skips; plain-text only
  • pg_restore — selective, reorderable, parallel (-j) restore from an archive-format dump
  • pg_basebackup — whole-cluster physical copy over the replication protocol; -R preps a new standby automatically
  • Base backup + archived WAL = point-in-time recovery; a logical dump cannot be combined with WAL replay
